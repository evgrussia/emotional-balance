# Risk Register: Emotional Balance (Нейро-Психолог 24/7)

## 1. Введение

Настоящий документ содержит реестр рисков проекта **Emotional Balance** -- ИИ-платформы психологической поддержки 24/7 через Telegram WebApp. Реестр охватывает все ключевые категории рисков: регуляторные, технические, бизнес-рыночные, финансовые, этические и операционные.

Реестр рисков является живым документом, подлежащим регулярному пересмотру и обновлению на всех фазах проекта.

### 1.1 Цели документа

- Идентификация и систематизация всех значимых рисков проекта
- Оценка вероятности и влияния каждого риска
- Определение стратегий митигации и планов contingency
- Назначение владельцев рисков и триггеров мониторинга
- Обеспечение основы для принятия решений и приоритизации

### 1.2 Методология оценки

Используется количественная шкала оценки по двум параметрам:

- **Likelihood (L)** -- вероятность наступления риска (1-5)
- **Impact (I)** -- влияние на проект при наступлении (1-5)
- **Risk Score (RS)** = Likelihood x Impact (1-25)

Классификация по Risk Score:

| Диапазон | Уровень | Действие |
|----------|---------|----------|
| 1-4 | Низкий (Low) | Мониторинг, без немедленных действий |
| 5-9 | Средний (Medium) | Активная митигация, план действий |
| 10-15 | Высокий (High) | Приоритетная митигация, эскалация |
| 16-25 | Критический (Critical) | Немедленные действия, блокер для фазы |

| Балл | Уровень | Описание | Частота |

## 7. Мониторинг и пересмотр

### 7.1 Частота пересмотра

| Действие | Частота | Ответственный |
|----------|---------|---------------|
| Полный пересмотр реестра | Ежемесячно | Business-Analyst Agent + Security Agent |
| Мониторинг Top-10 рисков | Еженедельно | Orchestrator Agent |

| Пересмотр после инцидента | Немедленно | Security Agent |
| Пересмотр при смене фазы проекта | При каждом Quality Gate | Orchestrator Agent |
| Обновление Risk Score после митигации | По факту внедрения | Risk Owner |

### 7.2 Процесс мониторинга

```
1. Проверить monitoring triggers для каждого риска
2. Обновить Likelihood и Impact при изменении обстоятельств
3. Пересчитать Risk Score
4. При RS >= 16 -- немедленная эскалация к пользователю
5. При RS 10-15 -- план действий в течение 1 недели
6. Зафиксировать изменения в журнале пересмотра
7. Обновить contingency plans при необходимости
```

## 8. Приложения

### 8.1 Глоссарий

| Термин | Определение |
|--------|------------|
| **Likelihood** | Вероятность наступления рискового события (1-5) |
| **Impact** | Степень влияния на проект при наступлении (1-5) |
| **Risk Score** | Количественная оценка = Likelihood x Impact |
| **Mitigation** | Меры по снижению вероятности или влияния |
| **Contingency** | План действий при наступлении риска |
| **Risk Owner** | Агент/роль, ответственный за мониторинг и митигацию |
| **Monitoring Trigger** | Индикатор материализации риска |
| **False Negative** | Пропуск реального события (кризиса) детектором |
| **False Positive** | Ложное срабатывание детектора |
| **BEP** | Break-Even Point -- точка безубыточности |
| **MAU** | Monthly Active Users |
| **LTV** | Lifetime Value -- пожизненная ценность клиента |
| **CAC** | Customer Acquisition Cost -- стоимость привлечения |
| **Burn Rate** | Скорость расходования средств (руб/мес) |
